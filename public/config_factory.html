Configuration File User Manual

This guide explains how to build a YAML configuration file that follows the specification.

The configuration controls the behavior of a team of members, each with its own LLM (Large Language Model) configuration, tools, and runtime options.

Top-Level Structure

Your YAML configuration must include at least:

welcome_message: Welcome to the team! # Optional

team_config: # Required
  <member_role>: <member_configuration>

workspace_files: # Optional
  <file_path>: <file_content>
  • welcome_message (string | null, optional): A message displayed at the start.
  • team_config (object, required): Defines all members of the team. Each key is the member’s name, and the value is its configuration.
  • workspace_files (object, optional): Dictionary of initial files to create in the workspace. Keys are file paths (relative to workspace root), values are file contents.

Member Configuration (MemberConfig)

Each member in team_config requires:

  • prompt (string | object, required): The initial prompt for the member.
  • llm (object, default: {provider: openai, model: gpt-4.1}): The LLM setup.
  • usage_limits (object, optional): Limits on model usage (requests, tokens, tool calls, …).
  • tools (array, optional): Tools the member can use.
  • max_help_requests (integer, default: 5): Max number of help requests and agent can do to other agents within a request (prevent the agent to loop with the same request).
  • hire_at_start (boolean, default: true): Whether to create at startup.

Example

team_config:
  researcher:
    prompt: Find and summarize information.
    llm:
      provider: openai
      model: gpt-4.1
      temperature: 0.7
    tools:
      - name: web_search
        max_results: 10
    max_help_requests: 3
    usage_limits:
      request_limit: 30
      tool_calls_limit: 50
    hire_at_start: true

Prompt

The prompt field can be specified in two ways:

1. As a String

Use a simple string as the prompt for the member:

prompt: You are a helpful assistant.

2. As an Object (with Template)

Use an object with a mandatory template field. The template can include placeholders in curly braces ({}) that will be replaced by the values of the other fields in the object.

All other fields in the object are available as replacement keys for the template. For example:

# Reusable field
_template: &TEMPLATE |
  You are a business analyst. Your specialty is {specialty}.

team_config:
  business analyst:
    prompt:
      template: *TEMPLATE
      specialty: "market research"

Special Parameters

  • member_name: Automatically replaced by the agent’s name (e.g., @Manager, @Developer)
    • Use {{member_name}} (double braces) in your template to reference the agent’s assigned name
    • Example: "You are {{member_name}}, a senior developer" becomes "You are @Developer, a senior developer"
  • team_profile: Allows to define in the agent profiles that will be aggregated under the key {team_profiles}

    • If a member’s prompt object contains a team_profile field, this value is aggregated across all members to produce a replacement key team_profiles.

    • In the template, use {team_profiles} to insert the aggregated list of all team profiles (with each member’s team_profile rendered and formatted with its own fields).

    • Example:

      # Reusable field
_template: &TEMPLATE |
  You are an helpful business analyst. Your specialty is {specialty}.
  Your are part of a team of :
  {team_profiles}

team_config:
  manager:
    prompt:
      template: *TEMPLATE
      specialty: coordinate project
      team_profile: - Manager, coordinates the team.
  analyst:
    prompt:
      template: *TEMPLATE
      specialty: finance
      team_profile: - Analyst, expert in {specialty}

      In this example, {team_profiles} in each template will be replaced by the concatenation of all members’ team_profile values, with their own fields substituted.

LLM Configuration

  • provider (string, default: openai, enum: openai, azure, nvidia, google-gla, mistral, anthropic)
  • model (string, default: gpt-4.1)
  • temperature (number | null, default: null): Controls randomness in responses (0.0 = deterministic, 1.0 = creative)
  • seed (integer | null, default: null): Random seed for reproducible outputs
  • max_tokens (integer | null, default: null): The maximum number of tokens to generate before stopping
  • openai_reasoning_effort (string | null, default: null, enum: low, medium, high): The reasoning effort level for OpenAI models that support it (e.g., o1 series)

Example

llm:
  provider: azure
  model: gpt-4.1
  temperature: 0.2
  seed: 123

Supported Providers and Models

You can use models from any of the supported providers:

OpenAI

  • provider: openai
  • models: See the OpenAI Pricing page for available models (e.g., gpt-4.1, gpt-4o, o1, etc.)
  • API key: OPENAI_API_KEY environment variable
  • Native structured output: ✅ Supported

Azure OpenAI

  • provider: azure
  • models: Same models as OpenAI, but deployed through Azure
  • API key: AZURE_OPENAI_API_KEY and AZURE_OPENAI_ENDPOINT environment variables
  • Native structured output: ✅ Supported

NVIDIA (OpenAI-compatible endpoint)

  • provider: nvidia
  • models: Access to NVIDIA’s open-source model inference API
    • Tested models:
      • mistralai/mistral-large-3-675b-instruct-2512 - Mistral Large 3 (675B parameters)
      • qwen/qwen3-235b-a22b - Qwen 3 (235B parameters)
    • See NVIDIA Build for all available models
  • API key: NVIDIA_API_KEY environment variable
  • Native structured output: ⚠️ Partial (models prefixed with openai only)
  • Note: NVIDIA Mistral models don’t support tool_choice='required' parameter (automatically handled)

Google Generative Language API

  • provider: google-gla
  • models: Google’s Gemini models via Generative Language API
    • Tested models:
      • gemini-2.5-pro - Gemini 2.5 Pro
      • gemini-3-pro-preview - Gemini 3 Pro (preview)
    • See Google AI Studio for all available models
  • API key: GOOGLE_API_KEY environment variable
  • Native structured output: ❌ Not supported (uses raw output_type with end_strategy='exhaustive')

Mistral

  • provider: mistral
  • models: Mistral’s native API models
    • Tested models:
      • mistral-large-latest - Latest Mistral Large model
    • See Mistral AI Docs for all available models
  • API key: MISTRAL_API_KEY environment variable
  • Native structured output: ❌ Not supported (uses raw output_type with end_strategy='exhaustive')

Anthropic

  • provider: anthropic
  • models: See the Claude Models page for available models (e.g., claude-3-5-sonnet-20241022, claude-3-opus-20240229, etc.)
  • API key: ANTHROPIC_API_KEY environment variable
  • Native structured output: ✅ Supported

Usage Limits (UsageLimits)

Limits on model usage to control costs and prevent runaway processes. These limits apply to each pydantic_ai run (ie request), not to the entire agent session.

Each limit can be set to null to disable it.

  • request_limit (integer | null, default: 50): Maximum number of requests to the model per LLM call
  • tool_calls_limit (integer | null, default: null): Maximum number of successful tool calls per LLM call
  • input_tokens_limit (integer | null, default: null): Maximum number of input/prompt tokens per LLM call
  • output_tokens_limit (integer | null, default: null): Maximum number of output/response tokens per LLM call
  • total_tokens_limit (integer | null, default: null): Maximum total tokens (input + output) per LLM call

Example:

usage_limits:
  request_limit: 30
  tool_calls_limit: 50
  total_tokens_limit: 100000

Tools

Web Search Tool (WebSearchTool)

Enables web searches to retrieve up-to-date information from the internet.

  • name: "web_search" (constant)
  • search (boolean, default: true): Enable web search functionality
  • crawl (boolean, default: false): Enable web crawler to recursively crawl websites
  • extract (boolean, default: false): Enable content extraction from web pages
  • max_results (integer, default: 5): Maximum number of web search results to return

Example:

- name: web_search
  search: true
  crawl: false
  extract: true
  max_results: 8

Vector Store Tool (VectorStoreTool)

Provides semantic search capabilities over indexed documents and files using three distinct tools:

  1. query_vector_store: Searches for relevant document chunks based on semantic similarity
    • Returns detailed chunks with content, score, metadata, and dates
    • Uses hybrid search (vector similarity + BM25 keyword matching, weighted 75% vector / 25% keyword)
    • Supports optional date filtering (year_from, year_to)
  2. search_file_names: Finds unique files related to a query
    • Returns file metadata only (name, file_id, file_path, dates) without content
    • Uses hybrid search weighted toward keywords (25% vector / 75% keyword) to better match filenames
    • Useful for discovering relevant documents before fetching full content
  3. fetch_google_drive_document: Retrieves the complete content of a specific document by file_id
    • Fetches all chunks for a file and reconstructs the full document
    • Returns complete document with metadata and full text content

Configuration Parameters

  • name: "vector_store" (constant)
  • database (string, default: "GoogleDrive"): The vector store database collection to query
  • max_number_files (integer, default: 25): Maximum number of results returned per search
    • For query_vector_store: limits the number of chunks returned
    • For search_file_names: limits the number of unique files returned
  • score_threshold (number, default: 0.5): Minimum similarity score for results (0.0 to 1.0)
    • Only chunks with scores ≥ threshold are returned by query_vector_store
    • Higher values (0.7-0.9) = more precise but fewer results
    • Lower values (0.3-0.5) = broader search with more results

Note: When using VectorStoreTool, all team members must use provider: azure for LLM configuration.

Example:

- name: vector_store
  database: GoogleDrive
  max_number_files: 50
  score_threshold: 0.7

Knowledge Graph Tool (KnowledgeGraphTool)

Provides access to a knowledge graph for storing and querying structured information.

  • name: "knowledge_graph" (constant)
  • read_only (boolean, default: false): If true, prevents modifications to the knowledge graph

Example:

- name: knowledge_graph
  read_only: true

Workspace Tool (WorkspaceTool)

Provides access to workspace files and code with two distinct execution modes:

File-based Tools (no container required)

  • file_view: Read-only text file access (read_text_file)
  • file_all: Full file operations (file_extractor, read_text_file, save_test_file)
    • file_extractor uses GPT-5-mini to intelligently extract and summarize information from files

SWE Tools (requires Docker container)

  • swe_view: View-only workspace access (view files, file tree, search)
  • swe_analysis: Code analysis tools (swe_view + create file & repo map)
  • swe_all: Complete workspace access with bash execution (swe_analysis + bash_command & edit file)

The SWE tools run inside a Docker container, enabling safe bash command execution and isolated file operations. File-based tools operate directly without containerization.

Configuration Parameters

  • name: "workspace" (constant)
  • tools (string, default: file_view): The tool set to enable

Example:

- name: workspace
  tools: swe_analysis

Workspace Files

The workspace_files field allows you to pre-populate the workspace with initial files when the team is created. This is useful for providing configuration files, documentation, templates, or any other files that agents need to access.

Configuration

  • workspace_files (object, optional): A dictionary where:
    • Keys are file paths relative to the workspace root
    • Values are the string contents of each file
    • Files are created automatically when the workspace is initialized
    • Nested directories are created automatically as needed

Example

workspace_files:
  README.md: |
    # Project Documentation
    This workspace contains project files.

  config/settings.json: |
    {
      "version": "1.0",
      "environment": "development"
    }

  scripts/setup.sh: |
    #!/bin/bash
    echo "Setting up environment..."

In this example:

  • README.md is created at the workspace root
  • config/settings.json is created in a config/ subdirectory (directory is auto-created)
  • scripts/setup.sh is created in a scripts/ subdirectory (directory is auto-created)

Use Cases

  • Configuration files: Provide default settings or environment configurations
  • Documentation: Include project documentation, guides, or instructions
  • Templates: Pre-populate templates for agents to use
  • Sample data: Include sample or seed data files
  • Scripts: Provide utility scripts or automation tools

Complete Example

welcome_message: Hello, team! Let's get started.

workspace_files:
  README.md: |
    # Team Workspace
    This workspace is shared by all team members.

  config/project.json: |
    {
      "name": "My Project",
      "version": "1.0.0"
    }

team_config:
  assistant:
    prompt: Answer questions clearly.
    llm:
      provider: openai
      model: gpt-4.1
      temperature: 0.8
    tools:
      - name: web_search
        max_results: 5
        search: true
      - name: vector_store
        database: GoogleDrive
        max_number_files: 20
        score_threshold: 0.5
      - name: workspace
        tools: swe_analysis
    max_help_requests: 5
    usage_limits:
      request_limit: 30
      tool_calls_limit: 50
      total_tokens_limit: 100000
    hire_at_start: true
